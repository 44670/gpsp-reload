# gpSP RV32 JIT Roadmap

- Status: active roadmap
- Measurement snapshot: 2026-07-18, benchmark IDs `rv32im-perf-m0-v2`,
  `rv32im-armwrestler-fast-ram-v2`, `rv32im-mips-frontend-v1`, and
  `rv32im-gba-memory-diff-v1`
- Scope: the native RV32IM backend in `riscv/` and its qemu-user development
  harness in `tests/rv32im/`

This roadmap turns the working RV32IM correctness proof into a measured,
gameplay-capable, production-quality gpSP dynarec. It complements
`docs/rv32im_backend_design.md`: the design note describes the backend
contract and accumulated proof surface, while this document defines the order
of work, performance methodology, milestone gates, and long-term completion
criteria.

The roadmap is intentionally evidence-driven. A smaller generated block is not
automatically faster, a qemu-user pass is not a hardware performance result,
and a synthetic fixture is not a substitute for a real frontend workload.

## North Star

The RV32 JIT is complete when real GBA workloads run through
`cpu_threaded.c`, execute generated native RV32 code without unexplained
interpreter fallback, match interpreter-visible behavior, and meet a declared
frame-time and audio deadline on a named RV32 target.

The final proof must include all of the following:

- Interpreter parity for guest registers, CPSR/SPSR and banked state, selected
  RAM/VRAM/IO state, scheduler events, and deterministic frame/audio output.
- Native execution evidence: native blocks and operations executed, generated
  bytes, bounded ROM-range guest-PC traces, branch-chain statistics, and an
  auditable fallback breakdown.
- Warm and cold performance measurements, with translation cost separated from
  steady-state execution.
- Native hardware measurements using verified architectural retired
  instructions and cycles, plus
  frame-time percentiles and audio-underrun counts.
- Correct executable-cache synchronization, branch patching, RAM-code
  invalidation, alert handling, IRQ/HALT behavior, and long-run cache stability.
- Stable automation that fails on timeout, mismatch, missing native evidence,
  unexplained fallback, performance regression, or missing external test data.

## Non-Negotiable Backend Contracts

Every optimization must preserve these contracts:

1. `cpu_threaded.c` remains the dynarec frontend authority. Backend changes may
   optimize its contract but must not silently redefine it.
2. `cpu.cc` and `gba_memory.c` remain the guest-semantics truth sources.
3. `mips/mips_emit.h` and `mips/mips_stub.S` remain the primary structural
   model for a mature native backend. RISC-V instruction selection and psABI
   details must still be designed specifically for RV32.
4. The baseline ISA remains plain RV32IM. The first production profile must not
   require `C`, bitmanip, floating point, vector, or vendor extensions.
5. ROM and RAM translation caches remain separate, and RAM code remains
   invalidatable. A performance shortcut may not bypass SMC handling.
6. Scheduler return, `update_gba()` flags, translation gates,
   `idle_loop_target_pc`, IRQ/HALT alerts, BIOS/SWI entry, and mode transitions
   are observable behavior.
7. Generated and patched code must receive correct instruction-cache
   synchronization on qemu-user and native hardware.
8. Fallbacks must be counted by source and explained. A fallback-only or
   interpreter-dominated run may not be presented as JIT success.
9. Successful harness commands use `result=PASS command=...`; failures use
   `result=FAIL command=...`. Synthetic and fixture-backed paths remain
   explicitly labeled.
10. ESP32-S3 is an Xtensa target. RV32 work must not add JS-backend work or be
    presented as CoreS3 SE performance evidence.

metric is `executed_rv32_insns`, sourced from QEMU
`in_asm,exec,nochain` translation-block traces. It is not labeled as an
architectural retired-instruction count.

`make -C tests/rv32im perf` and
`make -C tests/rv32im armwrestler-perf`; the latter streams QEMU trace data

### Current execution shape

- ARM `r0-r14` are fixed-mapped to RV32 host registers and packed NZCV is kept
  live in another mapped register.
- Direct patched branches can chain generated blocks without a C lookup while
  the cycle budget permits.
- Safe AL, no-flags, no-PC, unshifted register forms of common ARM ALU
  operations emit directly from mapped sources to the mapped destination.
  Flag-producing, carry-consuming, shifted, and PC-related forms retain their
  separately proven lowering paths.
- Ordinary EWRAM/IWRAM reads use shared RV32 leaf stubs for unsigned/signed
  byte, halfword, and word access, including mirroring and ARM unaligned-load
  semantics. The emitted site spills only dirty caller-saved mappings and
  invalidates them lazily; callee-saved mappings remain live.
- A non-RAM read tail stores the guest PC and enters the complete standard-ABI
  C memory helper. Dynamic ROM-range addresses take a two-instruction quick
  reject before the exact RAM-region test; when frontend constant analysis
  proves a non-RAM address, a two-instruction PC-relative call skips dispatch
  entirely. Writes and complex/stateful helpers retain the conservative full
  boundary.
- The cycle budget is currently stored in the JIT stack frame and loaded/stored
  at checkpoints.
- Block memory operations can cross the helper boundary once per transferred
  word.
- General Thumb coverage extends beyond the acceptance path but is not yet
  proven fallback-free across representative games.
- qemu-user provides strong ISA, ABI, frontend, and semantic evidence, but the
  harness still contains synthetic or runtime-fixture paths outside the real
  Armwrestler result gate.

## Measurement Contract

Performance work starts with measurement. No optimization milestone may use
only wall-clock anecdotes or generated-code size as its success criterion.

### Required measurement phases

Every performance workload must report two phases separately:

- `phase=cold`: reset caches, translate the workload, synchronize executable
  memory, and execute it for the first time.
- `phase=warm`: reuse translated blocks and measure only steady-state execution
  after a declared warm-up interval.

Translation time, cache synchronization, and execution time must not be merged
into a single unexplained number.

### Required RV32 metrics

The qemu-user performance gate probes `rdinstret` around bounded control and
workload windows, but it may report a `retired_insns` metric only after the CSR
has been shown repeatable and validated against an independent instruction
source. On the locked QEMU 10.0.8 linux-user build that condition is false, so
the raw/net CSR samples remain `rdinstret_csr_*` diagnostics with
`rdinstret_csr_verified=0`. The regression metric is instead
`executed_rv32_insns`, derived from QEMU `in_asm,exec,nochain` traces between
explicit markers, with `counter_source=qemu_exec_trace` and
`counter_semantics=qemu_tb_instruction_sum`. Wall time and an unverified CSR
must never be relabeled as retired work.

Each workload summary must include:

- `guest_insns` and `guest_cycles`
- `executed_rv32_insns` and `executed_rv32_insns_per_guest_x100` on qemu-user;
  `retired_insns` is reserved for a verified architectural counter
- generated block count and generated bytes
- translation count and translation/cache-sync bytes
- direct-chain attempts and hits
- fallthrough and indirect lookup counts
- scheduler/update exits
- helper calls by type and width
- mapped-register flush, store, invalidate, and reload counts
- fallback totals and source breakdown
- stable guest-state, memory-event, scheduler-event, and trace hashes

`rdcycle` and qemu-user wall time may be printed for diagnostics, but they are
not qemu-user regression gates. QEMU TCG timing is host-dependent and does not
model a specific native RV32 cache or pipeline.

### Required native-hardware metrics

After a hardware target is named, the same workload protocol must additionally
report:

- architectural cycles and retired instructions
- cycles and instructions per guest instruction
- branch misses and instruction/data cache misses when the PMU exposes them
- translation-cache flushes and peak cache occupancy
- frame-time p50, p95, and p99
- missed frame deadlines and audio underruns
- sustained performance and thermal state during a long run

The hardware performance target must be recorded per named CPU, clock, cache
configuration, memory configuration, compiler, and firmware/OS build. A MHz
number without those fields is not a valid baseline.

### Standard workload set

The performance harness should grow a small, stable workload set:

| Workload | Purpose |
| --- | --- |
| `mapped_alu` | Register-to-register ALU and multiply lowering |
| `flags_shift` | NZCV production, carry input, and ARM shift edge cases |
| `memory_read` | Aligned/unaligned byte, halfword, and word loads by region |
| `memory_write` | RAM stores, alerts, IO, OAM/palette, and SMC paths |
| `block_memory` | LDM/STM, push/pop, writeback, PC/SPSR cases |
| `branch_chain` | Internal/external direct chains and cycle exhaustion |
| `indirect_lookup` | BX, return-heavy code, ARM/Thumb switching, lookup hits |
| `scheduler` | Update refill, frame completion, IRQ, HALT, and idle loops |
| `mixed` | A representative cross-boundary sequence with no reset |
| `armwrestler` | Real frontend correctness and native-coverage gate |
| `full_frame` | Real frontend video/audio/input progression |

Every synthetic workload must end in a state hash checked against an
interpreter or explicit reference model. Performance is never allowed to trade
away semantic comparison.

### Example machine-readable performance line

```text
result=PASS command=rv32im-perf workload=mapped_alu phase=warm backend=rv32im guest_insns=32768 guest_cycles=32768 executed_rv32_insns=85409 executed_rv32_insns_per_guest_x100=260 counter_source=qemu_exec_trace counter_semantics=qemu_tb_instruction_sum rdinstret_csr_verified=0 helpers=128 fallbacks=0 state_hash=0x89018a9d reason=state_equal
```

Field names become compatibility contracts once a gate consumes them. New
fields may be appended; existing meanings should not be silently changed.

## M0: Trusted Performance Baseline

Goal: make dynamic RV32 cost measurable before changing instruction selection.

Deliverables:

- Add a deterministic `make -C tests/rv32im perf` target.
- Add cold and warm variants of `mapped_alu`, `memory_read`, `branch_chain`,
  `scheduler`, and `mixed`.
- Probe user-mode `rdinstret`, validate its semantics, and keep it
  diagnostic-only unless reliability is proven; otherwise use the explicitly
  named QEMU trace metric, never wall time.
- Measure and report the empty/control-window overhead.
- Add helper, mapped-state synchronization, chain, lookup, scheduler, and
  fallback counters behind a test/perf instrumentation configuration where
  production overhead would otherwise be material.
- Pin state, memory, scheduler, and trace hashes for every workload.
- Freeze guest instruction counts, workload definitions, expected hashes,
  Clang/QEMU versions, and RV32IM build flags in separately hashed workload
  and environment manifests.
- Record the baseline output and Makefile ceilings.
- Synchronize stale current-status and code-size statements in
  `docs/rv32im_backend_design.md`.

Exit criteria:

- Two consecutive clean runs produce identical instruction counts and hashes.
- qemu-user wall time is not used as a pass/fail criterion.
- Perf instrumentation can be disabled without changing generated production
  code.
- Existing `runtime`, `mixed`, `smoke`, `armwrestler`, and full `test` gates
  remain green.
- The baseline harness and its regression parser are committed as one coherent
  semi-milestone before ALU optimization begins.

## M1: Direct Mapped-Register ARM ALU Lowering

Goal: remove avoidable scratch-register moves from safe common ARM operations.

Initial safe scope:

- AL condition after frontend normalization
- no PC source or destination
- no flags update
- no register-specified shift
- mapped sources and mapped destination
- register forms of `MOV`, `MVN`, `ADD`, `SUB`, `RSB`, `AND`, `EOR`, `ORR`,
  `BIC`, and `MUL`
- immediate forms are optional within this milestone and may be added only
  where RV32 immediate encoding is exact and profitable

The first implementation should emit directly to the mapped destination when
source/destination overlap is architecturally safe, then mark the mapped slot
valid and dirty without emitting a redundant store move.

Required edge coverage:

- `rd == rn`, `rd == rm`, and `rn == rm`
- all mapped guest registers, including caller- and callee-saved host mappings
- constant-propagation interaction
- a conditionally reached block after frontend condition normalization
- a helper or scheduler boundary immediately after a direct result
- partial-unsupported block discard

Exit criteria:

- Safe register `ADD`, `SUB`, `AND`, `EOR`, `ORR`, `MOV`, and `MUL` fixtures
  emit one RV32 instruction for the guest operation body.
- Warm `mapped_alu` executed RV32 instructions per guest decrease by at least 25%
  from the M0 baseline.
- `mixed`, memory-control, and scheduler workloads do not regress in the
  QEMU-trace instruction count or state hashes.
- ARM Armwrestler cumulative bytes are strictly below 86084 and every per-test
  ceiling is tightened to the new green result with modest headroom.
- Thumb code size and Thumb helper count do not regress.
- `observed_results=79`, `failure_mask=0x00000000`, `fallbacks=0`,
  `fallback_events=0`, and `execute_arm_calls=0` remain true.
- The full RV32IM test gate passes before the optimization is committed.

Flag-producing, carry-consuming, shifted, PC-related, and PSR-visible forms
remain separate follow-up changes. Each expansion of the direct path must add
its own exact-overlap and interpreter-parity cases.

Measured result (2026-07-18):

- The exact-encoding gate covers 13 representative lowering cases. The
  `cpu.cc`-versus-native semantic gate covers 15 cases: every mapped guest
  register, caller- and callee-saved host mappings, all three data-processing
  source/destination overlap shapes, `MUL` destination/source and
  source/source aliases, the common logical/arithmetic operations, and
  one-instruction `MUL`.
- Warm `mapped_alu` fell from 169889 to 85409 executed RV32 instructions for
  32768 guest instructions: 5.18 to 2.60 RV32 instructions per guest, a 49.72%
  reduction. Generated bytes fell from 3760 to 1120, a 70.21% reduction.
- `memory_read` remained exactly 222049 instructions and 2536 bytes;
  `branch_chain` improved from 87585 to 75297 instructions and from 2424 to
  2040 bytes; `scheduler` improved from 827425 to 821281 instructions and from
  68 to 56 bytes; `mixed` improved from 87393 to 79585 instructions and from
  1500 to 1012 bytes. All pinned state, memory, scheduler, and trace hashes
  remained unchanged.
- Armwrestler remained 79/79 with zero failure mask, fallbacks, fallback
  events, and `execute_arm` calls. ARM cumulative generated bytes fell from
  86084 to 85508; Thumb remained exactly 39964 bytes with zero helpers.
- The byte-identical-text Armwrestler A/B replay records both sides of the
  tradeoff: combined cold work increases 0.31%, combined warm work decreases
  0.06%, ARM warm work decreases 0.08%, and Thumb warm work is unchanged.

## M2: Lower-Cost Helper and State Synchronization

Goal: reduce the cost of crossing from generated code to memory and stateful C
helpers without violating the RV32 psABI or guest-state visibility.

Work items:

- Measure which mapped registers are dirty, stored, invalidated, and reloaded
  at every helper class.
- Classify helpers by the guest state they observe or mutate.
- Define an explicit JIT-private helper boundary. Candidate implementations
  include leaf assembly stubs, selectively preserving veneers, or helper code
  built with a documented fixed-register contract.
- Keep standard C ABI calls for complex stateful paths until their private ABI
  behavior has direct tests.
- Avoid flushing a mapped register that a helper cannot observe and avoid
  invalidating a mapping that a helper cannot change.
- Preserve all state across alerts, IRQ checks, mode changes, SWI, SPSR
  restore, frame completion, and interpreter fallback.

Exit criteria:

- The helper ABI and clobber/observation table are documented beside the code.
- Every helper class has an ABI execution test under qemu-riscv32.
- Warm `memory_read` and `memory_write` instruction counts improve by a ratcheted
  amount established from M0 profiling; a nominal first target is at least 20%
  for ordinary non-alerting RAM accesses.
- The same-work real-frontend `memory_read` comparison improves in the same
  direction from the frozen RV32 value of 151362 while retaining exact MIPS,
  guest-work, hash, scheduler, and fallback evidence.
- Helper-event order, PC observations, values, state hashes, and alert hashes
  still match the interpreter/reference path.
- ABI tests prove callee-saved state, stack alignment, return linkage, and
  nested scheduler/helper exits.
- Armwrestler and the full test gate remain green with no new fallback.

Measured pure-read result (2026-07-18):

| Read path | Guest-state visibility | Host-call contract |
| --- | --- | --- |
| EWRAM/IWRAM `u8/s8/u16/s16/u32` | No PC materialization on the leaf path; address/value semantics match `gba_memory.c` | Shared RV32 leaf stub uses `a0`, `t0-t2`; emitted sites spill dirty mapped `a3-a7` and lazily invalidate them because the same call may take the C tail |
| Runtime non-RAM slow tail | Stores the instruction's guest PC before any BIOS/open-bus or device logic | Tail-calls the complete standard-ABI `read_memory*` helper; callee-saved guest mappings remain live |
| Compile-time proven non-RAM read | Stores guest PC and retains full C memory semantics | Two-instruction PC-relative standard-ABI call skips only the known-failing RAM-region test |
| Stores and stateful/alerting helpers | Full state remains materialized according to the existing helper contract | Conservative flush/invalidate/reload path remains unchanged |

- The generic standard-C pure-read boundary reduces the locked microbenchmark
  warm `memory_read` result from 222049 to 199073 instructions (10.35%) and
  generated bytes from 2536 to 1684. It avoids unconditional callee-saved
  spills and eager caller-saved reloads without changing any pinned hash.
- The production EWRAM/IWRAM leaf path reduces the same-work real-frontend
  warm RV32 count from 151362 to 105218 (30.48%), exceeding the 20% target;
  MIPS remains exactly 58691, and guest work, state/memory/scheduler hashes,
  update counts, generated-cache reuse, and zero-fallback evidence remain
  locked. The automated same-work floor is ratcheted to 30%.
- The fixture gate and `make -C tests/rv32im gba-memory-diff` each cover 75
  width/address cases across EWRAM, IWRAM, mirroring, alignments 0-3, both
  fast-region boundaries, slow-tail values, and guest-PC visibility. The
  latter cross-compiles and links the production `gba_memory.c` read functions
  and production RV32 fast stubs into one qemu-user binary, repeats stdout
  byte-for-byte, and records manifest, source, runtime, and binary hashes.
- The selector-isolated Armwrestler/Thumbwrestler A/B remains 79/79 with zero
  fallback. ARM warm work improves 0.12%, Thumb improves 0.23%, the total
  improves 0.16% above the locked 0.10% floor, and every warm group improves.

This completes the first pure-read submilestone, not all of M2: a deterministic
`memory_write` workload, per-class stateful-helper ABI tests, and the alerting
store boundary are still required before M2 is complete.

## M3: Inline Safe Memory Fast Paths

Goal: keep common memory accesses inside generated code and route exceptional
regions or side effects to audited slow paths.

Implementation order:

1. Aligned EWRAM/IWRAM reads.
2. Byte and halfword EWRAM/IWRAM reads with exact sign extension.
3. Unaligned word-load rotation semantics.
4. ROM reads when the required page is already mapped.
5. EWRAM/IWRAM stores with correct RAM-code invalidation checks.
6. Carefully selected VRAM/palette/OAM paths after their mirroring and side
   effects have dedicated tests.

The slow path remains mandatory for BIOS protection/open bus, unmapped or
paged ROM, backup/RTC devices, IO, alerting stores, special video-memory
semantics, and any unproven address class.

Required evidence:

- Region-by-region interpreter comparison for aligned and unaligned accesses.
- Boundary-address tests at every region transition.
- Exact read/write event order and value hashes.
- SMC tests proving RAM code is invalidated before stale native execution.
- Alert tests for IRQ, HALT, OAM update, and cache flush.
- No direct path may suppress ROM paging or open-bus behavior.

Implemented read slice (2026-07-18):

- The allowlist is exactly address regions `0x02` (EWRAM) and `0x03` (IWRAM)
  for `u8`, `s8`, `u16`, `s16`, and `u32` reads.
- EWRAM uses an 18-bit mirror mask and IWRAM uses a 15-bit mirror mask plus
  the existing `iwram + 0x8000` data half; both backing arrays now carry an
  explicit four-byte alignment required by RV32 `lh`/`lw` leaf loads.
- Unsigned unaligned halfword and word reads apply the ARM rotate rule; signed
  odd halfword reads sign-extend the addressed byte, matching
  `read_memory16_signed()`.
- Every other region takes the complete C path. No ROM paging, BIOS/open-bus,
  IO, video-memory, backup, RTC, store, alert, or SMC behavior was inlined.

M3 remains in progress because safe writes, mapped-ROM reads, full region-edge
coverage in production memory tests, SMC invalidation, and alert paths are not
yet complete.

Exit criteria:

- A declared allowlist of fast regions and access shapes is enforced in code.
- Ordinary RAM/ROM helper calls fall substantially on representative workloads.
- Warm memory workloads improve in QEMU-trace executed instructions and in
  verified native-hardware retired instructions/cycles once hardware exists.
- No memory, scheduler, or SMC parity regression appears in the full gate.

## M4: Block Memory, Swap, and Complex Memory Paths

Goal: remove repeated helper-boundary overhead from instructions that perform
multiple memory transfers.

Work items:

- Optimize common LDM/STM and push/pop forms while preserving transfer order,
  address progression, writeback ordering, base-in-list behavior, and PC value
  rules.
- Preserve user-bank transfer semantics and SPSR restore for `LDM ... {pc}^`.
- Consolidate helper work where safe; never collapse observable IO or alert
  ordering.
- Optimize SWP/SWPB without losing read-before-write order, old-value results,
  SMC, IRQ, or HALT behavior.
- Keep rare or risky forms on a proven slow path until separately covered.

Exit criteria:

- `block_memory` covers every addressing mode, load/store direction, writeback,
  base-list relationship, PC case, and alert class used by the frontend.
- Common RAM push/pop and LDM/STM paths reduce helper transitions and
  QEMU-trace executed instructions by a ratcheted amount.
- Interpreter-visible ordering and scheduler events remain identical.
- Real Armwrestler hotspots shrink and execute fewer QEMU-trace instructions;
  native retired-instruction claims wait for a verified hardware counter.

## M5: Control Flow, Lookup, and Cycle Accounting

Goal: minimize work between native blocks while retaining exact scheduler
boundaries.

Work items:

- Add dynamic counters for direct-chain success, cycle-expired chain exits,
  indirect lookup hits/misses, and C dispatcher entries.
- Keep same-cache short patch sites where range is proven; retain conservative
  long patch sites everywhere else.
- Investigate an inline or assembly fast lookup for hot BX/return and ARM/Thumb
  switch paths.
- Evaluate fallthrough chaining where frontend ownership and invalidation rules
  make it safe.
- Profile the stack-backed cycle budget. Compare it against alternatives that
  reserve a live host register or selectively spill a less valuable guest
  mapping.
- Preserve idle-loop forcing, update refill, PC-change, frame-complete, IRQ,
  HALT, and translation-gate behavior.
- Test branch patching after cache movement, flush, repatching, and RAM
  invalidation.

Exit criteria:

- `branch_chain`, `indirect_lookup`, and `scheduler` have deterministic dynamic
  baselines and interpreter hashes.
- Hot direct branches execute without a C dispatcher entry while cycles remain.
- The selected cycle strategy wins on mixed real workloads, not only a cycle
  microbenchmark.
- Branch patching and instruction-cache synchronization pass qemu-user and
  native hardware stress tests.
- No scheduler-boundary or trace divergence is accepted for performance gain.

## M6: Broad and Dense Thumb Lowering

Goal: make Thumb-heavy games use native code predictably rather than relying on
the subset covered by Thumbwrestler groups 0-2.

Work items:

- Build an exhaustive Thumb decoder corpus over all 65536 encodings, separated
  into valid, reserved, undefined, and frontend-special forms.
- Differential-test every directly lowered form against the interpreter over
  seeded register, flag, and memory states.
- Complete MIPS-aligned Thumb ALU, shifts, high-register operations, loads,
  stores, block memory, branches, BL pairs, BX, SWI, stack, and PC-relative
  forms.
- Apply known-value and flag-liveness analysis only when all guest-visible CPSR
  consumers are represented.
- Track dynamic Thumb helper and fallback use per real workload.
- Reduce Thumb code density without requiring RV32C.

Exit criteria:

- Reserved/undefined encodings remain explicitly rejected or handled according
  to interpreter behavior.
- The selected real-ROM suite has zero unexplained Thumb fallback.
- Thumb state, memory, scheduler, and trace hashes match the interpreter.
- Thumb executed RV32 instructions per guest and generated bytes are covered
  by qemu-user ratchets; native retired instructions are added at hardware
  bring-up.
- ARM performance and code size do not regress as a side effect.

## M7: Wider Differential and Real-ROM Coverage

Goal: move from one external conformance ROM to a representative correctness
suite before declaring broad compatibility.

Coverage layers:

- Seeded single-instruction differential tests.
- Seeded multi-instruction blocks with internal/external branches.
- Randomized register, CPSR/SPSR, banked-mode, memory, and scheduler state.
- Self-modifying RAM-code tests and translation-cache flush stress.
- Open conformance/test ROMs with machine-readable result buffers.
- User-supplied real workloads selected for ARM-heavy, Thumb-heavy,
  memory-heavy, IRQ/DMA-heavy, BIOS/HLE, and self-modifying behavior.

Every external ROM must have a deterministic signal and a documented local
path or environment variable. ROM data must not be committed when licensing
does not permit it. Missing ROMs fail external acceptance targets clearly; they
must not silently turn the target into a synthetic pass.

Exit criteria:

- Each workload has interpreter and JIT summaries with stable hashes and native
  evidence.
- Timeouts, crashes, divergence, and fallback-only execution are explicit
  failures.
- Any allowed fallback has a named reason, bounded count, and issue/milestone
  owner.
- Differential failures preserve a minimized seed or block fixture.

## M8: Full Headless Frontend and Frame/Audio Proof

Goal: run the real emulator progression rather than only CPU result hooks and
runtime fixtures.

Deliverables:

- Wire qemu-user commands to real emulator RAM, IO, video, audio, input, and
  scheduler state.
- Keep `run` for frontend frame/video/audio progression and `cont` for CPU or
  debugger boundaries.
- Support deterministic input scripts, breakpoints, bounded traces, framehash,
  RGB565/base64 output, PNG artifacts, and audio hashes.
- Compare interpreter and RV32 JIT at selected checkpoints and final frames.
- Add a noninteractive multi-frame workload suitable for perf measurement.

Exit criteria:

- Interpreter and JIT produce matching declared frame, memory, scheduler, and
  audio hashes for the selected workloads.
- PNG output is generated from real emulator frames, not a fixture snapshot.
- Runs terminate with parseable PASS/FAIL/timeout results and never hang CI.
- Warm multi-frame performance metrics are available before hardware bring-up.

## M9: Native RV32 Hardware Bring-Up

Goal: prove correctness and sustained performance on a named native RV32IM
system.

Bring-up checklist:

- Record CPU, clock, caches, memory, OS/firmware, compiler, ABI, and executable
  mapping model.
- Prove psABI entry/exit and signal/interrupt safety.
- Implement and test the platform's required data-cache clean, instruction-cache
  invalidate, and/or `fence.i` sequence after emission and patching.
- Verify writable and executable aliases if the platform uses separate views.
- Test W^X transitions where required by the OS.
- Stress ROM/RAM cache allocation, flush, patch, and invalidation for long runs.
- Capture architectural cycles, retired instructions, cache/branch PMU events
  where available, frame-time percentiles, and audio underruns.

Only after the first clean native run should the project set a hardware speed
target. The target should be expressed as a frame deadline and audio stability
at a named configuration, not as an unsupported clock-frequency estimate.

Exit criteria:

- The full correctness suite passes natively with the same machine-readable
  summaries as qemu-user.
- Native JIT execution and cache synchronization survive long-run and
  self-modifying-code stress.
- The named target meets its declared frame-time percentile and audio-underrun
  budget on the selected workload set.
- Cold translation spikes and cache flushes remain within separately declared
  latency budgets.

## M10: Production Hardening and Optional ISA Profiles

Goal: maintain RV32IM correctness and performance over time and add optional
features only when measurements justify them.

Work items:

- Add tiered CI: fast synthetic tests, qemu-user runtime tests, external-ROM
  acceptance, perf ratchets, and scheduled hardware runs.
- Preserve deterministic seeds, benchmark compiler versions, and machine
  configuration metadata.
- Add long-run cache churn, SMC, patching, and fault-injection tests.
- Investigate RV32C as an optional emission profile for instruction-cache and
  fetch-bandwidth improvement. Keep RV32IM as the portable baseline.
- Consider optional bitmanip or platform-specific profiles only behind explicit
  capability checks and separate correctness/performance gates.
- Compare against the MIPS backend using the same guest workloads and metrics;
  compare structure, generated bytes, helper counts, and dynamic host work
  rather than mechanically comparing opcodes.
- Document debugging, trace capture, benchmark reproduction, and regression
  bisection procedures.

Exit criteria:

- Default RV32IM remains fully supported and independently tested.
- Optional ISA profiles never change guest-visible behavior and can be disabled
  at build time.
- Performance thresholds are ratchets tied to reproducible workloads.
- A clean checkout can reproduce every non-hardware gate with documented tools
  and external inputs.

## Performance Ratchet Policy

Performance thresholds should become stricter only after a green, reviewed
result. They should not be loosened in the same change that regresses unless
the commit documents a correctness requirement, shows the measured tradeoff,
and records a follow-up milestone.

For each optimization:

1. Capture the pre-change cold and warm baseline.
2. Make one coherent backend change.
3. Compare state/event hashes before comparing performance.
4. Report `executed_rv32_insns` and helper/lookup counts on qemu-user; report
   retired instructions only where an architectural counter is verified.
5. Report generated bytes separately.
6. Run the full relevant correctness gate.
7. Tighten exact fixture and aggregate ceilings with modest deterministic
   headroom.
8. Commit the optimization and ratchets together.

Code size, executed/retired instructions, and cycles are separate axes. An
optimization may intentionally increase one to reduce another, but the
tradeoff must be measured on a representative workload and recorded in the
commit message or design note.

## Principal Risks and Mitigations

### Static code size is mistaken for speed

Mitigation: require warm QEMU-trace executed-instruction counts and, on
hardware, verified retired instructions, cycles, and frame-time percentiles.
Keep byte thresholds as an instruction-cache/code-cache metric only.

### QEMU wall time is mistaken for native performance

Mitigation: gate qemu-user on explicitly labeled QEMU-trace instruction sums
and semantic hashes. Establish architectural cycle, retired-instruction, and
frame targets only on a verified named hardware target.

### Helper optimization hides guest-state changes

Mitigation: classify helper state observation/mutation, preserve a slow
standard-ABI path, and compare registers, memory events, alerts, scheduler
events, and traces at each boundary.

### Register mapping improves ALU but worsens helper traffic

Mitigation: measure mapped-state flush/reload traffic per workload. Evaluate
the cycle register and guest-register mapping as a whole rather than optimizing
one microbenchmark in isolation.

### Inline memory bypasses special behavior

Mitigation: use an explicit region/access allowlist and route everything else
to a proven slow path. Add boundary, open-bus, paging, IO, video-memory, alert,
and SMC tests before expanding the allowlist.

### Fallback masks incomplete coverage

Mitigation: require source-separated fallback counters and bounded event logs.
Real acceptance targets fail on unexplained fallback even when final state
matches.

### Code cache growth causes churn or I-cache regressions

Mitigation: report generated bytes, peak resident cache occupancy, flush count,
and hardware I-cache events. Keep RV32C optional until the baseline profile is
correct and measured.

### Performance instrumentation changes the result

Mitigation: keep production-disableable counters, measure control overhead, and
confirm representative results with native PMU counters that do not modify the
generated hot path.

## Immediate Execution Sequence

M0, M1, the corrected same-work MIPS baseline, the selector-isolated
Armwrestler gate, the linked-real-`gba_memory.c` differential, and the first
pure-read optimization semi-milestone are complete. The next three coherent
semi-milestones are:

1. **Measured write-helper boundary**
   - Add a deterministic `memory_write` cold/warm workload with RAM, IO,
     alert, and SMC event hashes.
   - Classify the exact guest state observed and mutated by each store class
     before changing its conservative standard-ABI boundary.
2. **Safe EWRAM/IWRAM store path**
   - Fast-path only proven non-alerting stores first, retaining mandatory RAM
     code invalidation and a complete C slow path.
   - Require interpreter ordering, alert, scheduler, and stale-code tests plus
     a real-frontend same-direction result before ratcheting.
3. **Mapped-ROM read and block-memory profile**
   - Measure already-mapped ROM reads separately from paging/open-bus cases.
   - Use the resulting helper and transfer counts to choose between a safe ROM
     read path and the first common LDM/STM consolidation.

No write, ROM, video-memory, or device-region fast path should be added without
its own allowlist and machine-readable semantic evidence.

## Ultimate Definition of Done

The RV32 JIT may be described as production-ready only when:

- It passes the real frontend interpreter-vs-JIT suite, including Armwrestler
  and broader deterministic ROM workloads.
- Native execution counters and ROM-range traces prove that hot execution is
  generated RV32 code.
- Unsupported and fallback paths are zero or explicitly allowed and bounded
  per workload.
- Full frame, audio, input, memory, and scheduler behavior matches declared
  interpreter/reference hashes.
- A named native RV32 target meets its declared sustained frame-time and audio
  budgets.
- Cold translation, cache churn, SMC, branch repatching, IRQ/HALT, and long-run
  execution remain correct and bounded.
- RV32IM remains the tested portable baseline, regardless of optional extension
  profiles.
- All gates and reproduction steps are documented and machine-readable.

Until those conditions are met, roadmap reports should distinguish precisely
between emitter correctness, fixture parity, external-ROM parity, qemu-user
native execution, measured ISA efficiency, native hardware correctness, and
gameplay performance.
